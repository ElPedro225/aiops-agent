## 0:30 — What I Built (1 min)
**Say:**
> "The project simulates three microservices — auth, payments, and orders.
> Every run generates telemetry metrics: CPU, latency, error rate, and memory.
> The agent scores each data point for anomalies, decides what to do,
> and either acts automatically or asks for human approval.
> All of this is exposed through a REST API and a live dashboard."

**Show:** GitHub repo → README → architecture section.

---

## 1:30 — Architecture (1.5 min)
**Say:**
> "The system has six layers that each have one job."

**Show:** `ARCHITECTURE.md` or the README architecture table while you speak each layer:

| | |
|---|---|
| `data_ingestion/simulator.py` | Generates synthetic telemetry using NumPy normal distributions |
| `anomaly_detection/detector.py` | Two-stage scorer: Z-score then Isolation Forest |
| `policy_engine/policy.py` | Routes each score to AUTO, CONFIRM, or ESCALATE |
| `actions/remediation.py` | Executes the decided action (safe stubs) |
| `notifications/notifier.py` | Pushes phone alert via ntfy.sh |
| `api/server.py` + `ui/dashboard.html` | FastAPI backend + live browser dashboard |

## 3:00 — AI Techniques (2 min)
**Say:**
> "There are three AI or statistical techniques doing the heavy lifting."

### Technique 1 — Z-score (Statistical)
> "First, a Z-score flags any metric that is more than 3 standard deviations
> from the service's baseline mean. This catches sudden spikes quickly
> with no training required."

**Show:** `anomaly_detection/detector.py` — the z-score calculation lines.

### Technique 2 — Isolation Forest (ML)
> "Second, an Isolation Forest model is trained on the reference data.
> It isolates anomalies by building random decision trees — data points
> that are easy to isolate get a high anomaly score.
> The two scores are combined into a final 0-to-1 anomaly score."

**Show:** the `IsolationForest` section of `detector.py`.

### Technique 3 — LLM Reasoning (Claude API)
> "Third, when ANTHROPIC_API_KEY is set, each incident gets a two-sentence
> SRE explanation generated by Claude Haiku. The model gets the service name,
> score, decision, and metrics — and returns a human-readable diagnosis.
> This is gated by an environment variable so the system degrades gracefully
> without a key."

**Show:** `llm/claude_reasoner.py` — the prompt construction.

## 5:00 — How the Agent Makes Decisions (1 min)
**Say:**
> "Every scored event passes through the policy engine.
> If the score is above 0.75 — high confidence — it goes AUTO: the agent acts immediately.
> Between 0.40 and 0.75 — moderate confidence — it goes CONFIRM: a human reviews it on the dashboard.
> Below 0.40 — low confidence — it goes ESCALATE: a ticket is created and a human owns it.
> The dashboard lets you approve or escalate CONFIRM incidents with one click,
> and the table updates instantly."

**Show:** dashboard — click a CONFIRM incident → show the confirm panel → click Approve.

## 8:00 — Assumptions and Future Ideas (1 min)
**Say:**
> "A few assumptions I made: the telemetry is synthetic — a real system
> would ingest from Prometheus or Datadog. The Isolation Forest is retrained
> on every run — a production agent would use a persistent model.
> Action stubs are safe placeholders — real integration would call kubectl,
> PagerDuty, or a cloud API.

> Future improvements I'd add:
> — A real Prometheus data source instead of the simulator.
> — Persistent Isolation Forest model that improves over time.
> — Slack or email fallback alongside ntfy.sh.
> — A Docker Compose file so the whole stack runs with one command.
> — Multi-tenant support so each service has its own threshold profile."

---

## 9:00 — Challenges and Lessons Learned (45 s)
**Say:**
> "The biggest challenge was Evidently — the drift library — crashing on Python 3.14
> due to a Pydantic v1 incompatibility. I replaced it with a scipy KS-test
> implementation that uses the exact same statistical method, which taught me
> to always have a fallback for third-party dependencies.

> The second challenge was the dashboard's confirm buttons — SQLite stores booleans
> as integers, so JavaScript strict equality was silently failing.
> That taught me to always normalize types at the API boundary.

> Overall the biggest lesson: layered architecture makes debugging fast —
> each module has exactly one job, so failures are easy to isolate."
